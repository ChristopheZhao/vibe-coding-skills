## 1. Requirement Analysis
- User intent:
  - 为“经验沉淀”新增一个独立 skill 设计方案，先评审方案，再实施。
  - 该 skill 面向跨项目复用的经验/先验知识沉淀，不是项目运行时模块。
  - 触发模式以“用户主动触发”为主，可选“系统建议保存”，但必须低干扰、低误报。
- Scope in:
  - 明确 skill 定位、价值边界、触发与反触发规则。
  - 定义经验条目最小 schema 与写入/复用流程。
  - 明确与 `layered-project-memory` 的正交关系与引用协作方式。
  - 定义实施节奏与评审关口（先设计验收，再代码实施）。
- Scope out:
  - 不实现运行时评分引擎。
  - 不实现 orchestrator/harness 级事件监听系统。
  - 不把该 skill 设计成项目状态机或编排决策模块。
- Acceptance criteria:
  - 形成可评审的 skill 设计方案（定位、规则、结构、流程、验收标准完整）。
  - 方案明确“低 FP 通过规则约束达成，而非评分计算功能”。
  - 方案明确与 `layered-project-memory` 无语义重叠，仅引用关联。
  - 用户评审通过后，才进入实施计划执行。

## 2. Functional Decomposition
- 模块 A：定位与边界
  - 定义 skill 的独特价值（可复用经验与先验引导）。
  - 定义不该承担的职责（运行时策略引擎/系统功能）。
- 模块 B：触发与交互协议
  - 定义主动触发、建议触发、反触发条件。
  - 定义“是否保存经验”的单次询问与确认约束。
- 模块 C：经验条目模型
  - 定义最小经验卡字段与证据引用规范。
  - 定义去重、冷却、拒绝后静默等行为约束（规则层）。
- 模块 D：协作与复用
  - 定义与 `layered-project-memory` 的引用关系。
  - 定义后续在设计/review场景中的复用入口。

## 3. Implementation Approach
- Data flow:
  - 对话收敛节点 -> 命中触发规则 -> 询问是否保存 -> 用户确认 -> 经验卡写入 -> 后续检索复用。
- API/contract changes:
  - 新增 skill 目录：`experience-capture`。
  - 新增 `SKILL.md` 与 `references/` 合同文档（触发规则、经验卡模板、流程协议）。
  - 可选新增轻量脚本 `scripts/exp_ops.py`（create/list/link），仅做确定性辅助。
- Migration or compatibility notes:
  - 与现有 `layered-project-memory` 解耦，保持独立读写语义。
  - 仅通过 `source_event_refs` / `experience_refs` 做弱关联，不复制正文。

## 4. Technical Solution
- Key design choices:
  - `manual-first`：用户主动触发为主路径。
  - `suggest-with-hard-gates`：建议保存只在硬条件满足时触发，不使用评分引擎。
  - `confirm-before-write`：必须用户确认后写入。
  - `pointer-first`：证据与来源以引用为主，不复制长文本。
  - `orthogonal-design`：经验沉淀与项目状态记忆分层。
- Trade-offs:
  - 优先低打扰与高精确率，接受一定漏召回。
  - 优先规则可解释性，放弃早期复杂自动化能力。
  - 优先模板一致性，减少自由文本导致的质量漂移。
- Risks and mitigations:
  - 风险：提示过多打扰用户。  
    缓解：硬条件触发 + 单次询问 + 冷却规则。
  - 风险：与 `layered-project-memory` 重叠。  
    缓解：明确定义“项目状态 vs 复用经验”边界与引用关系。
  - 风险：经验卡变成流水记录。  
    缓解：强制包含 `decision_rules` 与 `review_checklist`。
  - 风险：场景过拟合（只服务 MAS 案例）。  
    缓解：以抽象模板设计，MAS 仅作为首个样例。
