## 1. Requirement Analysis
- User intent:
  - 仓库虽以 Codex 为主，但需要兼容其他 coding agent 使用。
  - 不应把 OpenAI 专有配置（`agents/openai.yaml`）提升为通用硬约束。
- Scope in:
  - 重定义仓库通用 skill 契约（SKILL-first，平台适配可选）。
  - 调整结构检查与 smoke 约束，移除对 OpenAI 元数据的硬依赖。
  - 调整发布脚本，明确多平台发布 profile（codex/claude/cursor/gemini/copilot）。
  - 补充兼容矩阵文档，明确核心契约与平台适配边界。
- Scope out:
  - 不实现各平台完整功能对齐（仅完成目录契约与发布兼容基线）。
  - 不重写现有 skill 的业务逻辑与语义边界。
- Acceptance criteria:
  - `check-structure.sh` 通过且不再强制 `agents/`。
  - skill smoke 不再强制 `agents/openai.yaml` 必需。
  - `release.sh`/`release-all.sh` 支持并校验目标平台集合（含 copilot）。
  - 新增兼容矩阵文档并在 README 中可发现。

## 2. Functional Decomposition
- Module A: 通用仓库契约
  - README 与结构检查脚本收敛为 SKILL-first。
- Module B: 多平台发布
  - 发布脚本工具 profile 与默认运行时路径策略统一。
- Module C: 兼容文档与回归
  - 兼容矩阵 + smoke 约束修订 + 基线验证。

## 3. Implementation Approach
- Data flow:
  - `skills/<name>/SKILL.md` 作为核心入口。
  - `agents/*` 作为平台可选适配层随发布一起分发，但不再强制存在。
- API/contract changes:
  - 仓库 contract 由「4目录必需」改为「`SKILL.md` 必需，`agents/references/scripts/assets` 可选」。
  - 发布工具 `--tool` 扩展为 `codex|claude|cursor|gemini|copilot`。
- Migration or compatibility notes:
  - 旧 skill 若有 `agents/openai.yaml` 可继续保留。
  - 无 `agents/` 的新 skill 也应可通过结构检查与基础发布。

## 4. Technical Solution
- Key design choices:
  - 保留 `agents/` 目录作为适配位，不再作为通用硬约束。
  - 采用“核心契约检查 + 平台适配可选”策略，避免 OpenAI 绑定。
- Trade-offs:
  - 减少硬约束后，单平台 UI 元数据一致性需要通过各 skill 自检补齐。
- Risks and mitigations:
  - 风险：历史 smoke 仍假设 `openai.yaml` 必需。
  - 缓解：逐个修订 smoke，改为可选校验或平台条件校验。
