## 1. Requirement Analysis
- User intent:
  - 新增一个面向 vibe coding 的 `knowledge-refresh` skill。
  - 目标是当先验不足或知识可能过期时，指导 agent 进行联网查证。
  - 当前阶段先完成规划文档，不直接进入复杂实现。
- Scope in:
  - 明确 skill 定位与边界（与 `layered-project-memory`、`experience-capture` 正交）。
  - 规划 skill 目录结构、关键文档、关键脚本和 examples。
  - 定义首版触发思路为 guide/cue 驱动，不引入二次门控评分机制。
- Scope out:
  - 不做运行时触发评分引擎。
  - 不实现独立联网执行器或 orchestrator 级状态机。
  - 不承担经验沉淀和项目状态存储。
- Acceptance criteria:
  - 形成可执行的中文计划，包含目录结构、内容契约、实施步骤和验收项。
  - 计划对“低打扰”和“不做重系统”有明确约束。
  - 用户确认计划后再进入 skill 文件实施。

## 2. Functional Decomposition
- Module A: Skill 定位与触发语义
  - 设计 `SKILL.md` 的 purpose、activation cues、do-not-use 场景。
  - 明确“用 guide 和 examples 驱动触发”，不做复杂规则计算。
- Module B: 参考文档与示例体系
  - 规划 `references/` 下边界说明、来源优先级、检索流程、输出契约、正反例。
  - 保证 examples 覆盖“应触发”和“不应触发”。
- Module C: 最小脚本与仓库集成
  - 规划 `scripts/smoke.sh`，仅做结构和文档契约 smoke。
  - 接入仓库 `.devtools/smoke.sh` 分发机制。
- Module D: 质量回归
  - 规划回归案例，防止 skill 滑向 memory、经验库或本地调试工具。

## 3. Implementation Approach
- Data flow:
  - 用户问题进入 -> skill 命中激活 -> 依据 guide 判断是否需要外部查证 -> 按 source priority 检索 -> 输出统一 verdict。
- API/contract changes:
  - 新增目录：`skills/knowledge-refresh/`
  - 新增文件：`SKILL.md`、`agents/openai.yaml`、`references/*`、`scripts/smoke.sh`
  - 仓库级改动仅涉及 `docs/plans` 和 skill 新目录，不改现有 skill 逻辑。
- Migration or compatibility notes:
  - 与已有 `experience-capture`、`layered-project-memory` 并存，不替代、不复写。
  - 保持当前 `.devtools/smoke.sh` 可插拔约定：每个 skill 自带 smoke。

## 4. Technical Solution
- Key design choices:
  - `guide-first`: 以说明和示例驱动，而不是引入触发算法引擎。
  - `lightweight`: 首版脚本只做 smoke，不做业务执行逻辑。
  - `source-first`: 强调外部证据来源优先级与时间有效性。
  - `orthogonal`: 明确与 memory/experience skill 的职责边界。
- Trade-offs:
  - 优先低复杂度和可维护性，接受首版自动化能力有限。
  - 优先高精度触发语义，接受一定漏召回。
- Risks and mitigations:
  - 风险：与其他 skill 语义重叠。  
    缓解：在 `positioning-boundary.md` 写死职责边界和反例。
  - 风险：触发提示过宽导致误触发。  
    缓解：`activation-cues.md` 增加反触发样例和非目标场景。
  - 风险：后续演进成重系统。  
    缓解：首版禁止引入评分门控与运行时状态存储。
